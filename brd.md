# Robot OS Lite BRD

> 更新时间：2026-04-25
> 适用范围：`robot_factory/` 当前 monorepo
> 文档定位：描述产品背景、目标用户、业务边界和阶段目标；详细实现与验收以 `prd.md` 为准

## 1. 产品背景

`Robot OS Lite` 面向 K12 教育、开发调试和内部交付场景，目标是把“手机端控制机器狗”这件事做成一套可复用、可部署、可扩展的控制系统，而不是一组一次性的 demo 脚本。

当前仓库已经完成了协议层、机器人端多传输运行时、Flutter SDK 和演示级 App 的主体骨架，并在 BLE 基础链路上完成了真实搜索、连接和数据交互验证。产品阶段已经从“从零设计”进入“把技术闭环收敛成可交付控制产品”。

这意味着后续工作的重点不再是重新设计大架构，而是：

- 把现有技术闭环整理成可稳定交付的产品能力
- 把演示级 App 推进到可实际操作的控制入口
- 把 BLE / MQTT / ROS 真机联调沉淀为稳定性和部署能力

## 1.1 长期产品规划

产品长期规划分为五条主线：

- 互联：手机 App、BLE、TCP、MQTT、机器人端运行时与 ROS 的连接能力
- 动作控制：姿态控制、连续运动控制、腿部参数控制、步态控制
- 教育：面向 K12 的图形化编程与教学场景
- 视觉：通过外置摄像头实现人物识别、手势识别与跟随
- SKILL 平台：让用户通过平台自定义动作和组合能力，而不是持续依赖研发手工新增动作

当前阶段判断：

- 互联：已形成可用主链
- 动作控制：已完成基础控制，正在向参数化控制扩展
- 教育：进入简单图形化编程 MVP 定义阶段
- 视觉：属于后续长期规划，不纳入当前 MVP
- SKILL 平台：属于后续长期规划，不纳入当前 MVP

## 2. 产品定位

### 2.1 一句话定位

一套面向机器狗的轻量控制系统，提供统一协议、统一 SDK、统一机器人端运行时，以及可演示、可扩展的移动端控制入口。

### 2.2 当前产品形态

系统由四部分组成：

- `apps/robot_app`：Flutter 控制 App，当前为演示级控制台 + 动作编排入口
- `mobile_sdk`：Flutter / Dart 统一 SDK，收敛连接、命令、状态和连接状态
- `robot_server`：Ubuntu 20.04 + ROS1 Noetic 机器人端服务，接 BLE / TCP / MQTT / ROS
- `protocol`：Python + Dart 共享二进制协议实现

## 3. 目标用户

### 3.1 核心用户

- K12 教育场景中的教师、学生
- 机器狗二次开发者和算法联调人员
- 内部测试、实施和交付人员

### 3.2 非目标用户

- 面向大众消费市场的普通宠物机器人用户
- 需要完整账号体系、云运营后台和商业化订阅能力的 SaaS 用户

## 4. 核心使用场景

### 场景 A：近距离控制

用户在机器人附近，通过手机 BLE 连接机器狗，完成连接、站立、坐下、停止、动作触发和基础运动控制。

### 场景 B：局域网调试

开发或测试人员通过 TCP 在同一局域网或 USB 网络环境下对机器人做调试、联调和问题定位。

### 场景 C：远程观察与远程控制

通过 MQTT 接入 broker，实现远程控制、状态同步和结构化事件观测。

### 场景 D：动作编排演示

用户在 App 中配置一个动作序列，顺序执行 `move / stand / sit / stop / do_action / do_dog_behavior`，用于课堂演示、脚本测试或标准动作复现。

### 场景 E：图形化肢体控制教学

教师或学生在图形化编程界面中，通过 block 组合去控制机器狗姿态和步态，例如：

- 坐下姿态
- 三腿着地
- 调整单腿位置、速度、角度等参数
- 调整双腿联动参数
- 调整四腿步态参数，以及 `roll / pitch / yaw / 跨度 / 速度` 等

### 场景 F：真机部署与验收

实施或测试人员在 Ubuntu 20.04 + ROS1 Noetic 机器人环境中启动 `robot_server`，接入 ROS topic，验证 BLE、TCP、MQTT 和状态采集链路。

## 5. 产品目标

### 5.1 当前阶段目标

1. 提供统一、稳定的控制协议和 SDK 接口，避免 App、transport、机器人端各写一套逻辑。
2. 支持 BLE / TCP / MQTT 三种链路，其中 BLE 是默认主链路，TCP / MQTT 是旁路调试和远程扩展链路。
3. 让 App 能完成基础连接、状态查看、快捷动作控制，以及面向 K12 的简单图形化控制入口。
4. 让机器人端可在 ROS1 Noetic 真机环境中部署，并将控制命令稳定映射到 ROS。

### 5.2 下一阶段目标

1. 把 App 从“演示控制台”推进到“可交付控制产品”。
2. 完成设备管理、配置持久化、正式手动控制 UI 和更清晰的错误恢复。
3. 增加面向 K12 的简单 block 编程能力，支撑基础教学演示。
4. 把 BLE 长时压测、多终端回归和 smoke 自动化补齐，降低真机联调成本。
5. 补齐协议 golden vectors、runtime 集成回归和 SDK 队列回归，压实跨端一致性。
6. 在后续阶段推进视觉能力，包括人物识别、手势识别和跟随。
7. 在后续阶段推进 SKILL 平台，让用户可通过平台自定义动作。

## 6. 当前已实现能力

### 6.1 协议与传输

- 已实现统一二进制协议：`0xAA55 | Type | Seq | Len | Payload | CRC16`
- 已支持 `CMD / STATE / ACK`
- 已有 Python / Dart 两套协议实现、CRC16 和 stream decoder
- MQTT topic 已固定为：
  - `robot/{id}/control`
  - `robot/{id}/state`
  - `robot/{id}/event`

### 6.2 机器人端

- 已实现 BLE / TCP / MQTT transport
- 已实现 ACK、重复包去重和 10Hz STATE 广播
- 已实现 ROS1 `/cmd_vel` 控制桥
- 已实现 ROS skill bridge，支持 `do_action` / `do_dog_behavior`
- 已实现 ROS 状态采集桥，能接电池、IMU、里程、诊断
- 当前尚未形成面向产品用户的自定义动作平台能力

### 6.3 SDK

- 已实现 `RobotClient.connectBLE()` / `connectTCP()` / `connectMQTT()`
- 已实现 `move()` / `stand()` / `sit()` / `stop()`
- 已实现 `doAction()` / `doDogBehavior()`
- 已实现命令队列、ACK 超时重试、BLE 扫描、连接状态流

### 6.4 App

- 已实现 BLE 扫描与连接
- 已实现 TCP / MQTT 连接配置弹窗
- 已实现状态面板与连接状态展示
- 已实现首页快捷动作控制
- 已实现动作编排的编辑、执行、暂停、恢复、停止
- 当前尚未实现面向课堂的 Blockly / 积木式图形化编程界面

## 7. 当前缺口

以下能力尚未完成，且是下一阶段的核心工作：

- App 缺设备绑定、最近设备记忆和更系统的配置持久化
- App 缺摇杆 / 连续运动控制等正式手动控制界面
- App 错误恢复仍偏调试风格，缺更明确的全局状态和恢复入口
- 缺面向 K12 的简单 block 编程工作台
- 缺 SKILL 平台，当前用户不能通过平台自定义动作
- SDK 尚未完成真正的 BLE > TCP > MQTT 自动回退
- BLE / MQTT / ROS 仍需要更多真机稳定性回归，不能视为端到端产品化完成
- MQTT / BLE smoke 还未沉淀为 CI 级自动回归

## 8. 关键业务前提

新增的 K12 教学能力和后续 SKILL 平台有一个共同前提：机器人底层必须暴露稳定、可复用的控制接口。

当前仓库已经验证的能力是：

- `move`
- `stand / sit / stop`
- `do_action`
- `do_dog_behavior`

在长期规划里，无论是教育还是 SKILL 平台，都不应绕过这些底层接口做“假能力”。
## 9. 产品边界

### 9.1 本阶段纳入范围

- 机器狗控制与状态同步
- BLE / TCP / MQTT 三链路统一接入
- Flutter SDK 统一控制接口
- App 侧基础连接、基础控制、简单图形化编程
- ROS1 Noetic 机器人端部署
- 面向 K12 的简单图形化编程 MVP

### 9.2 本阶段不纳入范围

- 用户账号体系
- 云端设备运营后台
- 多机器人编队控制
- 视频流、语音、地图导航等复杂上层能力
- 针对消费级用户的大规模产品化设计
- 视觉能力，包括外置摄像头人物识别、手势识别和跟随
- SKILL 平台，包括用户自定义动作平台与管理能力

## 10. 成功标准

从业务视角，当前阶段是否成功，主要看以下几点：

1. 新接入的 App 或调试工具不需要直接碰 transport 细节，只通过 `RobotClient` 就能完成连接和控制。
2. 机器人端能够在 Ubuntu 20.04 + ROS1 Noetic 环境下稳定部署，BLE 为默认主链路可用。
3. App 能完成连接、状态查看、快捷动作和动作编排演示，不再只是纯脚本验证。
4. 图形化编程若要交付，必须能映射到真实可执行的机器人底层控制接口，而不是只做 UI 模拟。
5. 当前 MVP 不应混入视觉和 SKILL 平台建设，避免范围失控。
6. 文档对“已实现 / 未实现 / 风险项”的描述保持诚实，不把 backlog 写成已交付。

## 11. 阶段判断

`Robot OS Lite` 当前不是从零开始的新项目，也不是已经可量产交付的成熟产品。

更准确的阶段定义是：

- 技术主链路已搭起
- 关键模块已可运行并具备真实联调基础
- 产品化工作集中在 App 体验、稳定性回归、部署规范和测试收敛

因此，后续研发应优先沿着“产品化收口”推进，而不是重新推翻现有架构。
