# Robot OS Lite BRD

> 更新时间：2026-04-28  
> 适用范围：`robot_factory/` 当前 monorepo 与三周 MVP 交付窗口  
> 文档定位：定义“可提供给用户使用的 App + 机器狗端全套软件”的业务目标、用户场景、交付边界和成功标准；具体排期见 `docs/mvp_schedule.md`

## 1. 背景与问题

当前项目已经具备协议层、机器人端服务、Flutter SDK 和演示级 App 的技术基础，并且 BLE 主链路已经完成真实搜索、连接和数据交互验证。接下来要交付的不是若干独立功能，也不是内部 demo，而是一套可以提供给用户实际使用的完整软件系统。

这套系统需要覆盖：

- 用户手机上的 App。
- 机器狗端常驻运行的软件服务。
- App 与机器狗之间稳定的 BLE 控制和状态同步。
- 机器人端与 ROS 的控制、状态采集和异常降级。
- 登录、设备绑定、升级包管理等云端支撑能力。
- 安装、部署、升级、回滚、验收和问题定位文档。

因此，本阶段的核心问题从“功能能不能做”转为：

1. 用户能否拿 App 连接并控制自己的机器狗。
2. 机器狗端服务能否长期稳定运行，并在异常后自动恢复。
3. 软件升级能否被安全触发、执行和回滚。
4. App 是否具备完整可用的产品入口，而不是开发调试控制台。
5. 交付团队是否能按文档安装、验收和定位问题。

## 2. 产品定位

### 2.1 一句话定位

`Robot OS Lite` 是一套面向机器狗的轻量控制与交互软件，提供用户可使用的移动 App、机器狗端常驻服务、BLE 控制链路、ROS 控制桥、状态同步、升级能力和基础账号绑定。

### 2.2 本阶段产品形态

本阶段交付的是“完整可用的 MVP 产品包”，而不是完整商业化平台。

产品包包含：

| 组成 | 说明 |
|---|---|
| 移动 App | 用户连接机器狗、切换模式、遥控、查看状态、使用 Face、手势、语音和图形化编程 |
| Flutter SDK | App 内部统一控制入口，收敛 BLE 连接、命令队列、ACK、状态流和错误流 |
| 机器狗端服务 | 常驻运行在机器狗 Ubuntu + ROS 环境，接 BLE、转 ROS、采集状态、执行升级 |
| 二进制协议 | App 与机器狗端之间的 CMD / STATE / ACK 统一协议 |
| 云端支撑 | 登录、设备绑定、升级包管理、升级触发接口 |
| 交付文档 | App 使用、机器狗部署、升级、回滚、BLE 联调、验收清单 |

## 3. 目标用户

### 3.1 核心用户

| 用户 | 需求 |
|---|---|
| 普通使用者 / 学生 | 打开 App，绑定机器狗，连接后能控制、互动和运行图形化程序 |
| 教师 / 教培人员 | 能在课堂演示机器狗动作、图形化编程、语音和手势交互 |
| 交付 / 实施人员 | 能安装机器狗端服务、确认服务运行、执行升级、定位基础故障 |
| 内部测试人员 | 能按验收清单做 BLE 长连、控制、状态、升级、回滚和 App 流程验证 |
| 二次开发人员 | 能基于统一 SDK 和协议扩展能力，不绕过主链路 |

### 3.2 非目标用户

- 需要完整云运营后台、复杂权限、订阅计费的商业 SaaS 用户。
- 需要远程实时控制、多机器人编队、视频流、地图导航的高级场景用户。
- 需要完整视觉平台、人物识别、跟随和开放动作平台的长期规划用户。

## 4. 产品目标

### 4.1 三周 MVP 业务目标

三周内交付一套用户可实际使用的 App 和机器狗端全套软件，满足：

1. 用户可以登录 App，绑定机器狗，并选择绑定设备进行连接。
2. App 可以通过 BLE 连接机器狗，并完成控制和状态查看。
3. 机器狗端服务可以守护运行，异常后自动恢复。
4. 机器人端可以通过 ROS 执行基础控制，并采集状态回传 App。
5. App 提供手机控制模式和狗头 Face 模式。
6. 狗头 Face 模式下自动开启手势控制和语音唤醒控制。
7. Face 表情由状态机驱动，能表达连接、监听、识别、执行、错误、低电量、升级等状态。
8. 用户可以使用基础图形化编程组合动作并执行。
9. 云端可以管理账号、绑定关系和升级包，触发机器狗端升级。
10. 交付团队可以按文档完成安装、升级、回滚和验收。

### 4.2 数据交互边界

本阶段 App 与机器狗之间的数据交互只支持 BLE。

```text
App
  -> mobile_sdk RobotClient
  -> BLE
  -> robot_server BLE service
  -> ROS 控制 / ROS 状态采集
  -> BLE STATE 回传 App
```

TCP / MQTT 可保留在既有代码中，但本阶段不作为用户产品链路，不纳入 MVP 开发、演示和验收。

## 5. 核心用户场景

### 场景 A：首次使用与绑定

用户打开 App，完成登录，扫描或输入机器狗信息，将机器狗绑定到账号。绑定完成后，App 展示已绑定设备列表，并允许用户选择默认机器狗。

成功结果：

- 用户知道当前账号绑定了哪台机器狗。
- 用户可以从绑定设备进入 BLE 连接流程。

### 场景 B：BLE 连接与手机控制

用户在机器狗附近打开 App，选择已绑定设备，App 通过 BLE 连接机器狗。连接成功后，用户进入手机控制模式，查看电量和姿态状态，并执行站立、坐下、停止、移动和动作调用。

成功结果：

- App 可以稳定连接机器狗。
- 控制命令能通过 BLE 到达机器人端，并由 ROS 执行。
- 机器狗状态能通过 BLE 回传 App。

### 场景 C：狗头 Face 互动

用户切换到狗头模式，App 展示 Face 表情，并自动开启手势识别和语音唤醒。用户通过语音或手势触发基础控制命令，Face 根据监听、识别、执行、错误等状态切换表情。

成功结果：

- 用户能直观看到机器狗当前交互状态。
- 语音和手势可以触发基础动作。
- 急停优先级最高，避免误触风险。

### 场景 D：图形化编程

用户在 App 中选择基础 block，例如前进、后退、左转、右转、站立、坐下、等待和动作调用，组成一个程序，保存后执行。App 按顺序通过 `RobotClient.*Queued()` 下发命令。

成功结果：

- 程序可以创建、保存、加载和执行。
- 执行顺序符合用户编排。
- 程序执行过程不绕过 SDK 和协议层。

### 场景 E：机器狗端服务运维

交付人员部署机器狗端服务，确认服务随系统启动，BLE 服务正常广播，ROS bridge 可工作。服务异常退出时，守护进程自动拉起。

成功结果：

- 机器狗开机后服务自动运行。
- 服务异常后可以自动恢复。
- 日志能帮助定位 BLE、ROS、升级问题。

### 场景 F：软件升级

云端发布升级包，交付人员或 App 触发升级。机器狗端下载 zip 包，校验、备份、替换代码并重启服务。如果升级失败，自动回滚上一版本。

成功结果：

- 升级成功后新版本生效。
- 升级失败后服务仍可恢复到旧版本。
- 升级过程有日志和状态可追踪。

## 6. 产品范围

### 6.1 本阶段必须交付

| 范围 | 说明 |
|---|---|
| App 登录与绑定 | 登录、token 保存、绑定机器狗、解绑、绑定列表、默认设备 |
| BLE 连接 | 扫描、连接、断线提示、重连、连接失败反馈 |
| 手机控制模式 | 基础遥控、动作控制、状态展示、急停入口 |
| 狗头 Face 模式 | Face 表情、状态机、进入后自动开启手势和语音 |
| 手势识别 MVP | 固定 3-5 个手势，映射基础控制命令 |
| 语音唤醒 MVP | 固定唤醒词和固定命令集合 |
| 图形化编程 MVP | 基础 block、保存、加载、执行、暂停、恢复、停止 |
| App UI 更新 | 主导航、双模式入口、核心页面视觉和交互一致 |
| 机器狗端守护 | systemd 常驻、异常重启、基础健康检查、日志 |
| ROS 控制与状态 | 基础控制发布、状态采集、异常降级、BLE STATE 回传 |
| 升级服务 | 云端下载 zip、校验、备份、替换、重启、失败回滚 |
| 云端最小支撑 | 登录、绑定关系、升级包管理、升级触发 API |
| 交付文档 | 安装、配置、升级、回滚、App 使用、BLE 联调、验收清单 |

### 6.2 本阶段不交付

- TCP / MQTT 作为用户数据交互链路。
- 多传输自动切换。
- 云端远程实时控制。
- 完整运营后台、角色权限、订阅计费。
- 完整视觉平台、人物识别、跟随。
- 完整 Blockly / 课程系统。
- SKILL 平台和用户自定义动作平台。
- 多机器人协同。

## 7. 业务成功标准

MVP 成功不是“功能代码合入”，而是用户和交付团队可以完成完整闭环。

### 7.1 用户侧成功标准

1. 用户可以安装并打开 App。
2. 用户可以登录并绑定机器狗。
3. 用户可以通过 BLE 连接绑定的机器狗。
4. 用户可以在手机控制模式下控制机器狗并看到状态。
5. 用户可以在狗头模式下使用 Face、手势和语音进行基础互动。
6. 用户可以创建并执行一个图形化程序。
7. 用户遇到断连、未连接、低电量、升级中、错误等状态时，App 有明确反馈。

### 7.2 机器狗侧成功标准

1. 机器狗端服务开机后自动运行。
2. BLE 服务能被 App 发现并连接。
3. ROS 控制链路能执行基础命令。
4. ROS 状态能通过 BLE 回传 App。
5. 服务异常退出后可以自动恢复。
6. 升级成功后服务重启并运行新版本。
7. 升级失败后可以回滚并保持可用。

### 7.3 交付侧成功标准

1. 交付人员可以按文档完成机器狗端安装。
2. 交付人员可以按文档完成 App 与机器狗绑定和连接验收。
3. 交付人员可以按文档触发升级和回滚测试。
4. 测试人员可以按清单完成 BLE 长连、断线重连、控制、状态、图形化编程、语音、手势和 Face 验收。

## 8. 关键约束

1. App 业务必须通过 `mobile_sdk.RobotClient` 使用控制能力，不直接依赖 BLE transport。
2. 图形化编程必须使用 `RobotClient.*Queued()`，保持 FIFO 顺序执行。
3. 手动控制、语音控制和手势控制使用实时控制语义，避免排队造成危险动作。
4. 急停优先级最高，任何模式下都不能被语音或手势覆盖。
5. 机器狗端必须兼容 Ubuntu 20.04 + ROS1 Noetic + Python 3.8。
6. App 与机器狗控制数据本阶段只走 BLE。
7. 升级服务必须先备份再替换，失败必须可回滚。
8. 文档必须诚实记录已完成、未完成和风险，不把长期规划写成已交付。

## 9. 主要风险

| 风险 | 影响 | 控制策略 |
|---|---|---|
| BLE 长连不稳定 | 用户无法稳定控制机器狗 | 从第 1 周开始做长连、断线、重连回归 |
| 升级失败导致服务不可用 | 机器狗端需要人工恢复 | 强制备份、校验、staging、重启验证和失败回滚 |
| 5 人配置同时做 App、后端、升级和联调 | 交付压力大 | 严格限制手势、语音、图形化编程范围，只做 MVP |
| App UI 更新挤占功能开发 | 功能无法完整闭环 | 先定 UI 基线，只覆盖核心页面，不做无边界视觉重做 |
| 手势和语音误触 | 可能触发非预期动作 | 固定命令集合、急停最高优先级、默认不执行高风险动作 |
| ROS 真机环境差异 | 本地测试无法覆盖 | 每周安排真机 smoke，不把真机联调堆到最后 |
| 云端接口延期 | 登录绑定和升级触发受阻 | 第 1 周冻结 API，App 先用 mock 并行开发 |

## 10. 阶段判断

当前项目不是从零开始，也不是已经成熟量产的完整商业平台。

本阶段的正确目标是：在现有技术主链路基础上，交付一套“用户可以安装、绑定、连接、控制、互动、编程、升级”的完整 MVP 软件包。

只要用户链路、机器狗端链路、云端最小支撑、升级运维和交付文档形成闭环，就可以认为本阶段 MVP 达成。视觉平台、SKILL 平台、多链路远程控制和复杂课程系统应进入后续版本规划。
