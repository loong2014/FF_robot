# 一个月 MVP 排期与整体开发规划

> 更新时间：2026-04-25
> 适用范围：当前 1 个月交付窗口
> 需求优先级：`prd.md` > `brd.md` > `docs/backlog.md` > 路线图文档 > `README.md`
> 文档定位：把当前已存在的技术闭环收敛成可交付、可演示、可回归的产品版本

## 1. 规划结论

当前仓库不是“从零开始”的项目。根据 `brd.md`、`prd.md`、`AGENTS.md` 与 `docs/backlog.md` 的交叉结论，当前阶段的关键问题已经从“把链路打通”转为“把已打通的链路产品化、稳定化和可验收化”。

因此，本月计划不再把基础连通性作为主目标，而是围绕以下四件事收口：

1. 把 `apps/robot_app` 从演示级控制台推进到可交付控制入口。
2. 把 `mobile_sdk` 收敛成单一、稳定、可测试的控制入口。
3. 把 `protocol` 和 `robot_server` 的跨端契约继续压实，减少漂移。
4. 把 BLE / TCP / MQTT / ROS 的真机联调沉淀为稳定性回归和交付文档。

## 2. 本月目标

本月交付的目标版本，应满足下面的产品边界：

- 互联：BLE 为默认主链路，TCP / MQTT 保留为调试和远程扩展链路。
- 动作控制：基础运动、站立、坐下、停止、已有动作调用可稳定执行。
- 教育：只做“简单图形化编程 / block 编排 MVP”，不做高级 Blockly 工作台。
- 产品化：具备设备管理、连接状态、错误恢复、最近设备记忆和可用的手动控制入口。

本月明确不纳入范围：

- 视觉能力
- SKILL 平台
- 云账号体系
- 多机器人编队
- 高级课程模板系统
- 复杂参数化动作编辑器的完整版本

## 3. 当前基线

根据现有文档和 backlog，当前基线可以概括为：

- `protocol` 已完成 Python / Dart 协议对齐，具备帧格式、CRC16 和 stream decoder。
- `robot_server` 已具备 BLE / TCP / MQTT transport、`StateStore`、10Hz STATE 广播、ROS1 运动控制桥（AlphaDog 默认 `/alphadog_node/set_velocity`）、ROS 状态采集桥和部署脚手架。
- `mobile_sdk` 已具备 `RobotClient`、命令队列、ACK 重试、连接状态和多 transport 接口。
- `apps/robot_app` 已具备 BLE 扫描、连接、状态面板、基础动作控制和动作编排入口，但仍偏演示级。

所以，本月的主线不是重做架构，而是把这些能力收口为“能交付给教师、开发者或实施人员使用”的版本。

## 4. 一个月排期

### 第 1 周：产品骨架收口

目标是先把“可用但分散”的能力收敛成单一产品入口。

| 工作项 | 模块 | 结果定义 | 验收口径 |
|---|---|---|---|
| 收敛 `mobile_sdk` 对外 export 边界 | `mobile_sdk` | App 只能通过 `RobotClient` 和必要模型访问能力 | App 不再绕过 SDK 直连 transport |
| 完成统一连接状态模型 | `mobile_sdk` | 连接状态、当前 transport、错误状态对 App 可见 | App 能稳定展示连接中 / 已连接 / 失败 |
| 完成设备管理基础页 | `apps/robot_app` | BLE 扫描、最近设备、手动选择、连接 / 断开入口 | 能在真机上找到设备并快速重连 |
| 补齐配置持久化基础 | `apps/robot_app` | 最近连接配置和设备记忆可落地 | 重启后配置不丢失 |
| 收敛协议回归测试 | `protocol` | Python ↔ Dart golden vectors 和基础解码回归补齐 | 字节级契约可复核 |

### 第 2 周：控制面落地

目标是把 App 从“演示控制台”推进到“真正可操作的控制面”。

| 工作项 | 模块 | 结果定义 | 验收口径 |
|---|---|---|---|
| 完成正式手动控制 UI | `apps/robot_app` | 摇杆 / 方向键 / 速度条 / 站立 / 坐下 / 停止 | 可直接驱动机器人做基础动作 |
| 收敛动作编排交互 | `apps/robot_app` | 动作序列编辑、执行、暂停、恢复、停止形成完整闭环 | 能演示一套固定动作脚本 |
| 落地简单 block 编程 MVP | `apps/robot_app` | 只做最小 block 容器和基础动作 block | 教师可拖拽并执行课堂演示 |
| 完成 block 到 `RobotClient` 的映射 | `mobile_sdk` + `apps/robot_app` | block 仅调用 SDK 公开接口 | 不绕过协议层和 transport 层 |
| 补齐全局错误提示与恢复入口 | `apps/robot_app` | 错误态、加载态、重试态可见 | 连接失败、参数错误、执行失败都有明确反馈 |

### 第 3 周：稳定性与自动化

目标是把“能跑”变成“能持续跑、可回归、可定位问题”。

| 工作项 | 模块 | 结果定义 | 验收口径 |
|---|---|---|---|
| BLE 长时压测与重连回归 | `robot_server` + `mobile_sdk` + `apps/robot_app` | 验证连续状态推送、断连重连和多终端回归 | 长时间运行不丢主链 |
| TCP / MQTT smoke 收口 | `robot_server` + `mobile_sdk` | 基础命令、ACK、STATE 路径可回归 | 调试链路不回退 |
| 运行时与 SDK 单测补齐 | `robot_server` + `mobile_sdk` | ACK、去重、队列、状态流、错误流回归可测 | 回归不依赖纯手工 |
| 协议契约继续压实 | `protocol` | golden vectors、粘包、CRC、坏包恢复更完整 | 跨端不漂移 |
| 设备状态与错误语义整理 | `mobile_sdk` + `apps/robot_app` | 连接态、错误态、恢复态定义更清晰 | UI 和 SDK 语义一致 |

### 第 4 周：验收与交付

目标是完成一次完整的产品交付闭环，而不是只停留在“代码差不多了”。

| 工作项 | 模块 | 结果定义 | 验收口径 |
|---|---|---|---|
| 真机联调与课堂演示回归 | 全模块 | 连接、控制、block 演示、断连恢复都跑通 | 一次完整流程可重复 |
| 修复遗留问题 | 全模块 | 仅处理影响交付的阻塞问题 | 不扩范围、不重构无关模块 |
| 更新部署与使用文档 | `docs` + `robot_server/README.md` + 根 `README.md` | 启动、依赖、联调、验收说明一致 | 新成员可按文档操作 |
| 形成交付清单 | 全模块 | 最终功能、限制、风险、后续计划明确 | 交付边界清楚 |

## 5. 模块级优先级

### `apps/robot_app`

本月最高优先级。

重点是设备管理、配置持久化、正式控制 UI、简单 block 编程 MVP、错误恢复和演示流程收口。

### `mobile_sdk`

第二优先级。

重点是 `RobotClient` 的边界收口、连接状态模型、错误流、自动切换扩展点和测试补齐。

### `protocol`

第三优先级，但必须持续推进。

重点是 Python / Dart golden vectors、解码边界、粘包 / 坏包恢复和字段约定一致性。

### `robot_server`

以稳定性和真机回归为主，不再做无关架构扩张。

重点是 BLE / TCP / MQTT / ROS 的稳定运行、烟雾测试、部署文档和真机联调。

## 6. 本月验收标准

本月结束时，应该能回答下面五个问题，并且答案是“能”：

1. App 是否可以通过 BLE 找到设备、记住最近设备并快速连接？
2. App 是否可以完成基础控制和简单 block 编排演示？
3. `RobotClient` 是否仍然是 App 唯一的控制入口？
4. 协议、SDK、机器人端的基础回归是否可自动验证？
5. 真机联调、部署和演示是否有完整文档支撑？

如果这五项不能成立，本月交付就不算完成。

## 7. 风险与控制

### 7.1 主要风险

- BLE / MQTT / ROS 的真机联调仍依赖真实设备和真实 broker，自动化程度有限。
- App 的产品化工作量大，容易把时间消耗在 UI 细节上。
- 协议、SDK 和机器人端如果不同步推进，容易出现跨端契约漂移。
- block 编程如果做得过大，容易滑向高级 Blockly 工作台，超出本月范围。

### 7.2 控制策略

- 以 `RobotClient` 为唯一控制入口，禁止 App 直连 transport。
- block 编程只做最小可演示版本，不引入复杂课程系统。
- 稳定性回归优先于功能扩张。
- 视觉和 SKILL 平台统一后移，不挤占本月交付容量。

## 8. 交付边界

本月计划完成的是“当前 MVP 的可交付版本”，不是完整产品线。

如果月内还有额外容量，才考虑补充以下增强项：

- BLE > TCP > MQTT 自动切换
- 更细的连接质量指标
- 更完整的主题样式和页面导航
- 更细的参数化动作编辑

这些增强项都不应影响本月主交付目标。
