# 三周 MVP 开发计划

> 更新时间：2026-04-28  
> 适用范围：2026-04-28 至 2026-05-18 三周 MVP 交付窗口  
> BRD 来源：`brd.md`  
> 数据链路范围：App 与机器狗之间只支持 BLE，暂不考虑 TCP / MQTT 等数据交互方案  
> 交付目标：完整可用的 App + 机器狗端 + 云端最小支撑 + 部署升级文档

## 1. MVP 目标

本 MVP 不是内部功能 demo，而是一套可以提供给用户实际使用的完整软件包。用户应能安装 App、登录、绑定机器狗、通过 BLE 连接机器狗、完成控制和互动；交付人员应能部署机器狗端服务、执行升级、回滚和验收。

### 1.1 本期交付的软件组成

| 组成 | MVP 交付内容 |
|---|---|
| 移动 App | 登录、设备绑定、BLE 连接、手机控制模式、狗头 Face 模式、手势/语音控制、图形化编程、核心 UI 更新 |
| Flutter SDK | `RobotClient` BLE 主链路、命令队列、ACK 重试、状态流、错误流、断线重连 |
| 机器狗端服务 | BLE 服务、ROS 控制桥、ROS 状态采集、守护运行、健康检查、日志、升级执行 |
| 云端最小支撑 | 登录、绑定关系、设备列表、升级包管理、升级触发 API |
| 协议 | BLE 承载的 CMD / STATE / ACK 二进制协议，保证粘包、CRC、ACK、状态回传稳定 |
| 交付资料 | App 使用说明、机器狗端部署文档、升级和回滚文档、BLE 联调文档、验收清单 |

### 1.2 核心链路

```text
用户
  -> App 登录 / 绑定设备
  -> App 选择绑定机器狗
  -> BLE 连接
  -> RobotClient 下发控制命令
  -> robot_server BLE service 接收
  -> ROS 控制机器狗
  -> ROS 状态采集
  -> BLE STATE 回传 App
```

本期明确不做 TCP / MQTT 产品化数据交互，不做 BLE > TCP > MQTT 自动切换，不做云端实时控制。云端只承担账号绑定、设备管理和升级包管理。

### 1.3 用户可感知的 MVP 能力

1. 用户可以登录 App。
2. 用户可以绑定机器狗、解绑机器狗、查看绑定列表并选择默认设备。
3. 用户可以通过 BLE 连接机器狗。
4. 用户可以在手机控制模式下完成基础遥控、站立、坐下、停止、动作调用和状态查看。
5. 用户可以进入狗头 Face 模式，看到表情状态。
6. 狗头模式下自动开启手势识别和语音唤醒控制。
7. 用户可以通过固定手势和固定语音命令触发基础动作。
8. 用户可以创建、保存、加载和执行一个基础图形化程序。
9. App 可以表达连接中、已连接、断开、执行中、错误、低电量、升级中等状态。
10. 机器狗端服务开机可自动运行，异常退出后可恢复。
11. 机器狗端软件可以从云端下载升级包并升级，失败可回滚。

### 1.4 本期不纳入范围

- TCP / MQTT 作为用户数据交互链路。
- 多传输自动切换。
- 云端远程实时控制。
- 完整运营后台、角色权限、订阅计费。
- 完整视觉平台、人物识别、跟随能力。
- 完整 Blockly / 课程系统。
- SKILL 平台和用户自定义动作平台。
- 多机器人协同。

## 2. 人员配置

本期按 5 人配置推进。5 人版本的原则是：优先完成用户可用闭环，严格限制手势、语音、图形化编程和 UI 更新范围，不做无边界扩展。

| 角色 | 人数 | 主要职责 |
|---|---:|---|
| 技术负责人 / SDK / 联调 | 1 | 排期与范围控制、`RobotClient` BLE 稳定性、队列语义、控制优先级、跨端联调、每周验收 |
| robot_server / ROS / 升级工程师 | 1 | 守护进程、BLE 服务稳定性、ROS 控制与状态、升级服务、真机部署和日志 |
| Flutter App 工程师 A | 1 | App UI 更新基线、主导航、手机控制模式、狗头 Face 模式、Face 状态机 |
| Flutter App 工程师 B | 1 | 图形化编程、账号绑定 UI、手势/语音模块接入、App 本地持久化 |
| 后端 / 云端 / 测试协调 | 1 | 登录绑定接口、设备绑定关系、升级包管理、升级触发 API、mock 服务、测试记录 |

### 2.1 5 人配置下的范围约束

- 手势识别只做固定 3-5 个手势，不扩展完整视觉平台。
- 语音唤醒只做固定唤醒词和固定命令集合，不做大词表和复杂语义理解。
- 图形化编程只做基础 block 和顺序执行，不做完整 Blockly / 课程系统。
- App UI 更新只覆盖核心页面和主流程，不追求全量视觉细节打磨。
- QA 不设独立岗位，由技术负责人牵头，每周固定组织真机验收。
- 后端只提供最小接口，不做完整运营后台。

## 3. 第 1 周：产品底座与交付底座

时间：2026-04-28 至 2026-05-04

### 3.1 本周目标

先建立“用户可用产品”的底座：BLE 主链路稳定、机器狗端服务可守护运行、云端最小接口冻结、App UI 和用户主流程定基线、升级服务跑通原型。

### 3.2 开发计划

| 工作项 | 负责人 | 结果定义 |
|---|---|---|
| 产品主流程确认 | 技术负责人 + App A + 后端 | 明确登录、绑定、选择设备、BLE 连接、控制、狗头模式、图形化编程、升级的端到端路径 |
| App UI 更新基线 | App A | 定义主导航、手机控制模式、狗头模式、图形化编程、登录绑定页面的布局和视觉基线 |
| 登录绑定接口冻结 | 后端 | 冻结登录、token、绑定、解绑、绑定列表、默认设备 API，提供 mock |
| App 登录绑定骨架 | App B | App 具备登录页、绑定列表页、默认设备选择的基础流程，可接 mock |
| BLE 主链路稳定性回归 | 技术负责人 | `RobotClient` 连接、ACK、状态流、错误流、断线提示和重连策略收口 |
| 机器狗端守护进程 | robot_server 工程师 | systemd 管理 robot_server，开机启动，异常退出自动重启 |
| BLE 服务健康检查 | robot_server 工程师 | 检查 BLE 服务是否启动、广播是否正常、是否能处理连接 |
| ROS 控制与状态基础回归 | robot_server 工程师 | 基础控制命令能到 ROS，ROS 状态能写入 StateStore 并经 BLE 回传 |
| 升级包规范 | robot_server 工程师 + 后端 | 定义 zip 包结构、manifest、版本号、校验字段、备份和 staging 路径 |
| 升级服务原型 | robot_server 工程师 | 跑通下载、校验、解压到 staging、备份、替换、重启的测试链路 |
| 第 1 周真机 smoke | 全员 | BLE 连接、基础控制、状态回传、服务重启、升级原型各跑通一次 |

### 3.3 本周验收

1. 用户主流程和 App 核心页面结构已经确认。
2. 后端登录、绑定、解绑、设备列表、升级触发接口已冻结，并能用 mock 调试。
3. App 可以进入登录和绑定流程，至少能使用 mock 完成绑定列表展示。
4. App 可以通过 BLE 连接机器狗，并完成基础控制命令下发。
5. ROS 控制链路能驱动机器狗执行基础动作。
6. ROS 状态能通过 BLE STATE 回传到 App。
7. robot_server 可由 systemd 拉起，手动 kill 后能自动恢复。
8. BLE 服务启动失败或异常退出时有可定位日志。
9. 升级服务可用测试 zip 跑通一次原型流程。
10. 第 1 周验收记录已形成，包括通过项、失败项和阻塞项。

## 4. 第 2 周：用户核心体验闭环

时间：2026-05-05 至 2026-05-11

### 4.1 本周目标

让用户可以实际使用 App 完成主要体验：登录绑定、BLE 连接、手机控制、狗头 Face 互动、手势和语音控制。机器狗端继续补齐升级回滚和 ROS/BLE 稳定性。

### 4.2 开发计划

| 工作项 | 负责人 | 结果定义 |
|---|---|---|
| App 主导航和 UI 基线落地 | App A | 登录、绑定、手机控制、狗头模式、图形化编程入口统一到可用导航 |
| 手机控制模式 | App A + 技术负责人 | BLE 连接状态、基础遥控、站立、坐下、停止、急停、动作调用、状态展示可用 |
| 狗头 Face 模式 | App A | 进入模式后展示 Face，页面可表达监听、识别、执行、错误、低电量、升级中等状态 |
| Face 状态机 | App A | 统一连接状态、语音状态、手势状态、执行状态，不让各模块直接改表情 |
| 手势识别 MVP 接入 | App B | 固定 3-5 个手势映射为基础控制命令，经 `RobotClient` 走 BLE 下发 |
| 语音唤醒 MVP 接入 | App B | 固定唤醒词 + 固定命令集合，经 `RobotClient` 走 BLE 下发 |
| 控制优先级收口 | 技术负责人 | 急停最高优先级；手动、语音、手势冲突时有明确规则 |
| 登录绑定真接口接入 | App B + 后端 | App 接入登录、绑定、解绑、绑定列表、默认设备接口 |
| 升级服务回滚 | robot_server 工程师 | 替换后重启失败时自动恢复上一版本 |
| BLE + ROS 稳定性增强 | 技术负责人 + robot_server 工程师 | 断线、重连、状态延迟、ROS 异常降级有明确反馈 |
| 第 2 周真机验收 | 全员 | 用户完整路径：登录、绑定、BLE 连接、手机控制、狗头互动跑通 |

### 4.3 本周验收

1. App 主导航、模式入口和核心页面符合 UI 更新基线。
2. 用户可以登录并看到绑定设备列表。
3. 用户可以绑定和解绑机器狗，并选择默认设备。
4. App 可以围绕默认设备发起 BLE 连接。
5. 手机控制模式可以完成基础遥控、站立、坐下、停止、急停和动作调用。
6. App 可以展示连接状态、电量、姿态和最近状态更新时间。
7. 狗头模式进入后自动启动语音监听和手势识别。
8. Face 表情能随连接、监听、识别、执行、错误、低电量和升级状态切换。
9. 至少 3 个手势可以通过 BLE 控制机器狗。
10. 至少 3 条语音命令可以通过 BLE 控制机器狗。
11. 急停优先级最高，语音和手势不会覆盖急停。
12. 升级失败场景可回滚到上一版本。
13. 第 2 周验收记录已形成，并明确第 3 周必须修复的问题。

### 5.2 开发计划

2. 用户可以选择默认机器狗并通过 BLE 快速连接。
3. 手机控制模式可以稳定完成基础控制、动作调用、状态查看和急停。
4. 狗头 Face 模式可以稳定展示状态表情，并自动开启手势和语音。
5. 手势和语音命令通过 BLE 控制机器狗，且不绕过 `RobotClient`。
6. 图形化编程可以创建、保存、加载并执行一个完整动作序列。

11. BLE 主链路通过 30-60 分钟长连测试。
12. ROS 控制和状态采集在长连期间没有明显卡死或崩溃。
13. App 核心页面 UI 已按本期基线完成收口，不遗留明显割裂的临时页面。

## 7. 风险与控制

| 风险 | 影响 | 控制策略 |
|---|---|---|
| BLE 长连不稳定 | 直接影响用户控制和状态同步 | 第 1 周开始长连测试，每周真机 smoke |
| 升级失败导致服务不可用 | 机器狗端需要人工恢复 | 强制备份、staging、重启验证和失败回滚 |
| 5 人配置压力大 | App、后端、升级和真机联调互相挤压 | 限制手势、语音、图形化编程和 UI 更新范围 |
| App UI 更新范围失控 | 功能闭环延期 | 只覆盖主导航和核心页面，不做全量视觉重做 |
| 手势 / 语音误触 | 可能触发非预期动作 | 固定命令集合，急停最高优先级，默认不执行高风险动作 |
| 云端接口延期 | 登录绑定和升级链路受阻 | 第 1 周冻结 API，App 使用 mock 并行开发 |
| ROS 真机环境差异 | 本地测试无法代表实际机器人 | 每周真机验收，问题当天记录和分级 |

## 8. 最终验收口径

三周结束时，需要完成一次完整用户流程演示：

1. 安装 App。
2. 登录账号。
3. 绑定机器狗。
4. 选择默认机器狗。
5. 通过 BLE 连接机器狗。
6. 在手机控制模式下执行基础控制并查看状态。
7. 切换狗头 Face 模式，使用手势和语音触发基础动作。
8. 创建并执行一个图形化程序。
9. 触发机器狗端升级，验证成功升级。
10. 触发失败升级场景，验证回滚。
11. kill robot_server，验证守护进程自动恢复。
12. 完成 BLE 长连和断线重连验收。

只有当 App、机器狗端服务、云端最小支撑、升级回滚和交付文档都形成闭环时，本期 MVP 才算完成。
