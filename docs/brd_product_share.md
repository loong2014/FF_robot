# Robot OS Lite — 产品与会议分享版

> 派生自仓库根目录 `brd.md`（2026-04-25），面向产品、教研、交付等非纯研发读者。  
> 技术细节与验收标准以 `prd.md` 为准。

---

## 1. 一句话与阶段判断

**一句话：**  
一套面向机器狗的轻量控制系统：统一协议、统一手机 SDK、统一机器人端服务，加上可演示、可扩展的移动端控制入口。

**我们现在在哪：**  
不是从零立项，也不是可量产成熟产品——**技术主链路已打通**（含 BLE 真机验证），当前重心是 **产品化收口**：把演示级 App 做成稳定可用的控制产品，并把真机联调沉淀为稳定性与部署规范。

---

## 2. 为什么要做（业务背景）

- 目标场景：**K12 教育**、**开发调试**、**内部交付**，而不是一次性 demo。
- 诉求：手机控机器狗这件事要变成 **可复用、可部署、可扩展** 的体系，避免各端各写一套、难以维护。
- 后续重点：**稳定交付**、**可操作的控制体验**、**真机链路上的稳定性与部署能力**。

---

## 4. 典型场景（便于对齐需求）

| 代号 | 场景 | 要点 |
|------|------|------|
| A | 近距离控制 | 手机 BLE：连接、站立/坐下/停止、动作、基础运动 |
| B | 局域网调试 | TCP：同网或 USB 网络下调试、联调、排障 |
| C | 远程 | MQTT：远程控制、状态、结构化事件 |
| D | 动作编排 | App 内配置序列，顺序执行多种指令，用于课堂演示与复现 |
| E | 图形化教学（规划） | Block 控制姿态、步态与参数（与真实底层接口对齐） |
| F | 部署验收 | Ubuntu + ROS 环境启动服务，验证各链路与状态采集 |

---

## 5. 系统长什么样（四块，不讲仓库路径）

1. **控制 App（Flutter）** — 当前偏演示控制台 + 动作编排；正向「可交付控制入口」演进。  
2. **手机 SDK** — 统一连接、命令、状态、连接状态，业务应通过 SDK 接入，而非各写各的传输层。  
3. **机器人端服务** — 接 BLE / TCP / MQTT 与 ROS，部署在目标机器人环境。  
4. **协议层** — 跨端统一的二进制协议与编解码，保证一致性。

---

## 6. 五条产品主线与「当前阶段」

| 主线 | 含义 | 当前判断 |
|------|------|----------|
| 互联 | App、BLE、TCP、MQTT、机器人运行时、ROS | **已形成可用主链** |
| 动作控制 | 姿态、连续运动、腿部/步态参数化 | **基础已有**，向参数化扩展 |
| 教育 | 图形化编程与教学 | **MVP 定义与推进中** |
| 视觉 | 外置摄像头：人、手势、跟随 | **长期规划，本阶段 MVP 不纳入** |
| SKILL 平台 | 用户自定义动作与组合 | **长期规划，本阶段 MVP 不纳入** |

---

## 7. 本阶段目标（可对齐排期）

**当前阶段：**

1. 统一、稳定的协议与 SDK，避免多端重复实现。  
2. **BLE 为主链路**，TCP / MQTT 为调试与远程扩展。  
3. App：连接、状态、快捷动作，以及面向 K12 的 **简单图形化控制入口**（与真实控制对齐）。  
4. 机器人端在目标 ROS 环境下可部署，控制命令稳定映射到 ROS。

**下一阶段（产品化）：**

- App：设备管理、配置持久化、正式手动控制 UI、清晰错误恢复。  
- K12：简单 block 编程，支撑基础教学演示。  
- 质量：长时压测、多终端回归、自动化 smoke，降低真机成本。  
- 测试与文档：协议 golden、runtime/SDK 回归，**诚实区分已实现 / 未实现 / 风险**。

---

## 8. 范围边界（会议里常要问的）

**本阶段做：** 机器狗控制与状态同步；三链路统一；统一 SDK；App 基础连接与控制、简单图形化编程 MVP；机器人端部署与 K12 简单图形化 MVP。

**本阶段不做：** 账号体系与云运营后台；多机编队；视频/语音/地图等复杂上层；消费级大规模产品设计；**视觉整条线**；**SKILL 平台整条线**。

---

## 9. 已具备能力 vs 主要缺口（产品语言）

**已具备（摘要）：**  
统一二进制协议（含校验与流式解析）；CMD / STATE / ACK；MQTT 主题规范；机器人端多传输、状态广播、ROS 运动与技能桥、状态采集；SDK 的连接与核心指令 API、队列与 ACK 重试；App 的 BLE/TCP/MQTT、状态面板、首页快捷动作、动作编排编辑与执行控制。

**主要缺口：**  
设备绑定与配置持久化；正式摇杆/连续运动等手动控制界面；面向用户的错误恢复与全局状态；K12 积木式编程工作台；SKILL 平台；多链路自动回退；真机稳定性与 CI 级自动回归仍需加强。

**共同前提（对教育与 SKILL 都成立）：**  
上层能力必须建立在 **稳定、可复用的底层控制接口**（运动、姿态、技能调用等）之上，不做「只有 UI、没有真控」的假能力。

---

## 10. 成功标准（业务视角）

1. 新 App 或工具 **只通过统一 SDK** 即可完成连接与控制，不直接绑死在某种链路上。  
2. 机器人端在目标环境下 **可稳定部署**，BLE 作为默认主链路可用。  
3. App 能完成连接、状态、快捷动作与编排演示，**脱离纯脚本验证形态**。  
4. 图形化若交付，必须 **映射到真实可执行** 的底层接口。  
5. **当前 MVP 不混入视觉与 SKILL 平台**，避免范围失控。  
6. 对外文档与沟通 **如实反映** 实现度与风险。

## 12. 建议 Q&A 备忘

- **和消费级产品区别？** 教育、开发、内部交付；不做账号云运营与大规模 C 端体验。  
- **为什么 BLE 是默认？** 近距离、低依赖，适合课堂与现场演示；TCP/MQTT 补调试与远程。  
- **视觉什么时候做？** 长期主线，本 MVP 明确排除，避免与产品化抢资源。  
- **和 PRD 关系？** 本文件讲「为什么、给谁、边界」；具体功能列表与验收以 PRD 为准。
